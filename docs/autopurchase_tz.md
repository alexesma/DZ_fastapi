# ТЗ: Автозаказ на пополнение товарных остатков

## 1. Цель

Внедрить в систему управляемый автозаказ на пополнение склада, который:

- рассчитывает потребность в пополнении по нашему наличию;
- предлагает поставщика и количество к заказу;
- формирует черновики заказов поставщикам;
- работает через очередь исключений и понятные причины рекомендации;
- не перегружает Dragonzap бессмысленными массовыми запросами;
- допускает поэтапный переход от ручного подтверждения к частичной автоматизации.

Документ фиксирует первую рабочую архитектуру под текущую кодовую базу.

## 2. Что уже есть в системе

### 2.1. Уже реализовано

- Аналитика по позиции и кроссам:
  - текущий остаток по нашему прайсу;
  - в пути;
  - расход за 30 дней;
  - фактический срок поставки;
  - `reorder_point`;
  - `optimal_order_qty`;
  - `draft_purchase_order`;
  - `ABC/XYZ`;
  - supplier stats;
  - очередь исключений.
- Старый pipeline пополнения:
  - `GET /order/generate_restock_offers`
  - `POST /order/confirm`
  - `GET /order/confirmed`
- Сущности для сохранения решения о пополнении:
  - `AutoPartRestockDecision`
  - `AutoPartRestockDecisionSupplier`
- Поля номенклатуры:
  - `minimum_balance`
  - `min_balance_auto`
  - `min_balance_user`

### 2.2. Ограничения текущей реализации

- Текущий `reorder_point` упрощён:
  - `avg_daily_decrease_30_days * average_actual_lead_days`
- Нет выделенного жизненного цикла запуска автозаказа:
  - кто считал;
  - когда считал;
  - по каким правилам считал;
  - какие строки были auto-approved, а какие нет.
- Старый `restock_offers` и новый `tracking/exceptions` живут параллельно и логически дублируют друг друга.
- Полный массовый site-запрос по всей номенклатуре в Dragonzap слишком дорогой по времени и нагрузке.

## 3. Бизнес-решение для первой версии

### 3.1. Главный принцип

Первая версия автозаказа строится на **детерминированных правилах**, а не на LLM.

ChatGPT допускается только как помощник:

- объяснить, почему система предлагает заказать;
- объяснить, почему позиция попала в исключения;
- подготовить текст письма поставщику;
- подготовить комментарий менеджеру.

ChatGPT не должен:

- сам рассчитывать количество к заказу;
- сам выбирать поставщика без правил;
- сам отправлять заказ поставщику.

### 3.2. Источник предложений поставщиков

Для фонового автозаказа в `v1`:

- потребность в пополнении считается по нашим данным;
- источник закупки и поставщик берутся **только с сайта Dragonzap**.

Локальные прайсы поставщиков:

- не используются как финальный источник закупки;
- могут использоваться только как вспомогательная аналитика в других экранах;
- не должны определять поставщика в автозаказе.

Причина:

- фактический заказ сейчас оформляется через сайт;
- менеджерам важнее видеть сразу боевой источник закупки;
- логика автозаказа должна совпадать с реальным каналом оформления.

Ограничение:

- чтобы не перегружать сайт, запросы в автозаказе выполняются только по строкам,
  где уже рассчитана реальная потребность в пополнении.

### 3.3. Рабочий режим

Первая версия автозаказа работает в трёх режимах:

1. `draft_only`
   - система только формирует черновики;
   - менеджер подтверждает вручную.

2. `auto_approve_safe`
   - безопасные позиции подтверждаются автоматически;
   - спорные уходят в ручную проверку.

3. `disabled`
   - для отдельных SKU, брендов или категорий автозаказ отключён.

По умолчанию запускать `draft_only`.

## 4. Что считаем для каждой позиции

### 4.1. Базовые входные данные

Для каждой позиции должны использоваться:

- текущий остаток в последнем нашем прайсе;
- позиции в пути;
- расход за 30 дней;
- расход за 90 дней;
- фактический срок поставки;
- fill rate поставщиков;
- текущие site-предложения поставщиков Dragonzap;
- `minimum_balance`, если он задан вручную;
- класс `ABC/XYZ`.

### 4.2. Как считаем спрос

Для `v1` использовать:

- `avg_daily_30 = sold_last_30_days / 30`
- `avg_daily_90 = sold_last_90_days / 90`
- `avg_daily_blended = 0.7 * avg_daily_30 + 0.3 * avg_daily_90`

Если 90-дневной истории нет:

- использовать только 30 дней.

Если 30-дневной истории нет, но есть ручной `minimum_balance`:

- использовать `minimum_balance` как fallback-политику.

### 4.3. Как считаем lead time

Для позиции берём:

1. фактический срок по выбранному поставщику, если есть достаточная история;
2. если истории мало, текущий срок из прайса;
3. если и этого нет, fallback по провайдеру или глобальный default.

### 4.4. Как считаем safety stock

Для `v1` использовать простую, объяснимую модель:

- `lead_time_demand = avg_daily_blended * lead_time_days`
- `safety_stock_days` определяется по классу `ABC/XYZ`
- `safety_stock = avg_daily_blended * safety_stock_days`

Рекомендуемые стартовые буферы:

- `AX`: 14 дней
- `AY`: 10 дней
- `AZ`: 7 дней
- `BX`: 10 дней
- `BY`: 7 дней
- `BZ`: 5 дней
- `CX`: 7 дней
- `CY`: 5 дней
- `CZ`: 0 дней

Для `CZ` автозаказ по умолчанию не включать.

### 4.5. Как считаем точку дозаказа

- `reorder_point = lead_time_demand + safety_stock`

### 4.6. Как считаем целевой остаток

Для `v1`:

- `review_period_days = max(7, lead_time_days)`
- `target_stock = reorder_point + avg_daily_blended * review_period_days`

### 4.7. Как считаем рекомендуемое количество к заказу

- `available_qty = current_quantity + in_transit_qty`
- `recommended_order_qty = max(ceil(target_stock - available_qty), 0)`

Если есть ручной `minimum_balance`, то:

- `target_stock` не должен быть меньше `minimum_balance`

Итог:

- `target_stock = max(target_stock, minimum_balance)`

### 4.8. Минимальная партия и кратность

Итоговое количество должно учитывать:

- `min_qnt`
- `multiplicity`

Правило:

- округлять `recommended_order_qty` вверх до допустимой кратности.

## 5. Как выбираем поставщика

### 5.1. Базовый принцип

Для вашего процесса цена важнее срока, поэтому поставщик выбирается как:

**самый дешёвый из допустимых**, а не поставщик с абстрактным общим score.

### 5.2. Фильтры допустимости

Поставщик считается допустимым, если:

- это не наш собственный прайс;
- есть текущая цена;
- есть положительное наличие или поставщик допускается к backorder;
- fill rate не ниже порога;
- срок не выходит за максимально допустимый для класса позиции.

Стартовые пороги:

- `fill_rate >= 80%` для auto-approve;
- `fill_rate >= 60%` для draft/manual review;
- для `AX/BX` допускаемый срок строже;
- для `CZ` auto-approve запрещён.

### 5.3. Сортировка кандидатов

Если поставщик прошёл фильтры, сортировать по:

1. цене;
2. fill rate;
3. фактическому сроку;
4. наличию.

### 5.4. Fallback

Если допустимых поставщиков нет:

- позиция попадает в очередь исключений;
- draft purchase order не формируется автоматически.

## 6. Какие позиции попадают в автозаказ

### 6.1. Базовый кандидат

Позиция попадает в расчёт, если выполняется хотя бы одно из условий:

- `available_qty < reorder_point`
- `estimated_days_left <= 14`
- есть ручной `minimum_balance` и текущий остаток ниже него
- позиция выпала из последнего нашего прайса, но по ней недавно был спрос

### 6.2. Исключить из auto-approve

Даже если позиция требует пополнения, auto-approve не делать, если:

- класс `CZ`;
- нет надёжного поставщика;
- позиция выпала из последнего нашего прайса;
- резкий рост цены;
- нет истории спроса и нет ручной политики;
- недостаточно данных по сроку и fill rate.

Такие позиции идут в `needs_review`.

### 7.1. Что показывать в строке

- расход 30/90;
- средний спрос в день;

- reorder point;

- цена;

### 7.2. Фильтры

## 8. Draft purchase orders

### 8.1. Что нужно на выходе

После расчёта система должна уметь группировать строки по поставщику и формировать черновики заказов:

- один поставщик = один draft order;
- внутри него несколько позиций.

### 8.2. Что должно быть в draft order line

- `autopart_id`
- бренд
- OEM
- наименование
- рекомендованное количество
- выбранный поставщик
- цена
- срок
- reason/explanation
- snapshot расчёта:
  - остаток
  - в пути
  - расход

  - ABC/XYZ

## 9. Что делаем со старым restock pipeline

### 9.1. Что сохранить

Сохранить сущности:

- `AutoPartRestockDecision`
- `AutoPartRestockDecisionSupplier`

### 9.2. Что изменить

Не использовать старый pipeline как основной расчётный механизм.

Вместо этого:

- считать новый автозаказ через `placed_orders.py` и новый сервис;
- старые эндпоинты пометить как legacy;
- постепенно перевести UI `/restock` на новый источник данных.

### 9.3. Что считать новой основной логикой

Новая основная логика должна жить отдельно, например:

- `dz_fastapi/services/autopurchase.py`

И использовать:

- данные собственного прайса;
- историю движения;
- историю заказов;
- supplier stats;
- текущие site-предложения Dragonzap.

## 11. Новые API

### 11.1. Расчёт

- `POST /autopurchase/runs`
  - запуск расчёта
- `GET /autopurchase/runs`
  - список запусков
- `GET /autopurchase/runs/{run_id}`
  - итог запуска
- `GET /autopurchase/runs/{run_id}/items`
  - строки расчёта

### 11.2. Работа менеджера

- `POST /autopurchase/runs/{run_id}/approve`

- `POST /autopurchase/runs/{run_id}/create-drafts`
  - сформировать draft purchase orders

### 11.3. Draft orders

- `GET /autopurchase/drafts`
- `GET /autopurchase/drafts/{draft_id}`
- `POST /autopurchase/drafts/{draft_id}/confirm`

## 13. Автоматизация и расписание

### 13.1. Ночной расчёт

Нужен scheduler job:

- рассчитывать автозаказ по нашему прайсу 1 раз в сутки;
- сохранять `autopurchase_run`;
- уведомлять менеджера о новых критичных позициях.

### 13.2. Автоподтверждение

В `v1` включать только для safe-сегмента:

- `A/B`
- не `Z`
- fill rate выше порога
- позиция не выпала из прайса
- нет ценовой аномалии
- есть корректный поставщик

Остальные строки должны идти в `needs_review`.

## 14. ChatGPT / OpenAI

### 14.1. Разрешённый scope

ChatGPT можно подключать только как explainability-слой:

- “Почему система предлагает заказать 18 шт?”
- “Почему выбран этот поставщик?”
- “Какие риски у этой строки?”
- “Подготовь письмо поставщику”

### 14.2. Что не отдавать LLM

Не отдавать модели право:

- финально выбирать количество к заказу;
- менять поставщика;
- автоматически отправлять заказ.

### 14.3. Технический формат

Рекомендуемый интерфейс:

- backend считает всё детерминированно;
- backend отдаёт в LLM snapshot строки;
- LLM возвращает:
  - `human_explanation`
  - `risk_summary`
  - `manager_note`
  - `supplier_email_draft`

## 16. Критерии готовности первой версии

Первая версия считается готовой, если:

- по последнему нашему прайсу формируется очередь автозаказа;
- для каждой строки видны:
  - остаток;
  - в пути;
  - спрос;
  - reorder point;
  - recommended qty;
  - выбранный поставщик;
  - причина выбора;
- можно отфильтровать `auto-approved / needs review / blocked`;
- можно сформировать draft orders по поставщикам;
- фоновые расчёты не гоняют сайт по всей номенклатуре без фильтра потребности;
- старый `restock_offers` остаётся доступным, но помечается как legacy.

## 17. Что начинаем делать сразу после утверждения ТЗ

Первый практический шаг:

1. создать `autopurchase.py`;
2. вынести в него расчёт строки автозаказа;
3. добавить новую схему ответа для очереди автозаказа;
4. сделать `POST /autopurchase/runs` и `GET /autopurchase/runs/{run_id}/items`;
5. только потом делать UI.

Это даст быстрый и управляемый старт без ломки текущего контура заказов.
